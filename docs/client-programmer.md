## 1. Creavas — 빙벽 등반 생존 게임 🏆 교내 게임잼 우승 (팀, 1박 2일)

| | |
| --- | --- |
| 개발 기간 | 2025.06.28 ~ 29 (1박 2일 게임잼) |
| 역할 | 게임 시스템 개발 (데이터 테이블, 맵툴, 아이템) |
| 기술 | Unreal Engine 5.6, Blueprint |
| 성과 | **Dangling 교내 게임잼 우승** |

"얼음"을 주제로 한 무박 2일 게임잼 참가작(4인 팀, **참가팀 중 유일한 언리얼 엔진 사용 팀**). 빙벽을 타고 올라가며 **체온(체력)이 모두 떨어지기 전에 목표 높이에 도달**하는 생존 게임입니다.

### 게임 메커니즘

- **'Getting Over It' 방식 발사형 이동** — 빙벽의 붙잡기 포인트를 활용해 상승
- 시간 경과에 따라 체온(체력 100)이 점진 감소 → 긴장감의 원천
- 아이템으로 리스크 관리: **핫팩**(일정 시간 체온 손실 방지), **피켈**(새 붙잡기 포인트 생성)

### 결정적 피봇

초기 기획은 단순한 플랫포머였지만, 빠르게 프로토타입을 만들어 본 결과 "너무 단순하다"는 결론이 나와 **'Getting Over It' 방식 발사형 이동으로 방향을 전환** — 게임성이 획기적으로 개선되며 우승으로 이어졌습니다. *명확한 방향성이 결과의 질을 좌우한다*는 교훈을 얻은 프로젝트입니다.

### 담당 구현과 잼 이후 확장

- 잼 기간: 게임 시스템 개발 — 아이템(핫팩/피켈)·체온 시스템, 난이도 구성
- **잼 이후**: 1회성으로 끝내기 아까워 확장성을 고려한 **데이터 테이블 + 엑셀 기반 레벨 에디터**를 추가 구현 — 기획자가 원하는 맵을 직접 만들 수 있는 구조로 발전 (실무의 테이블 데이터 시스템 경험 활용)

## 3. 폴리글롯 (MRTranslator) — AR 실시간 번역기 (캡스톤, 팀 5인)

| | |
| --- | --- |
| 팀 구성 | 5명 (캡스톤 6조) |
| 기술 | Unity, Meta Quest 3, Google Cardboard |
| 외부 서비스 | Naver Papago, Google ML Kit, Vito.ai ReturnZero, Google Cloud STT |

외국인 학생이 한국어 수업을, 한국 학생이 외국인 교수의 영어 수업을 따라가기 어렵다는 캠퍼스의 실제 문제에서 출발한 졸업 프로젝트.
**현재 상용 XR 기기 2종(Quest 3 + Cardboard)으로 증강현실 실시간 자막 번역**을 구현했습니다.

### 아키텍처 — 인터페이스 기반 SDK 추상화

```
음성 입력 ──▶ IAudioStreamWebService (STT)
              ├─ ReturnZero (한국어 특화 — 인식률로 채택)
              └─ Google Cloud STT (WebSocket 스트리밍, 다국어)
텍스트   ──▶ ITranslatorService (번역)
              ├─ Papago (메인 — 번역 품질로 채택)
              └─ Google ML Kit (온디바이스 폴백)
```

- **메인/폴백 자동 전환**: Papago 응답 실패 시 `TranslatorManager`가 온디바이스 ML Kit으로 자동 전환 — 네트워크 불안정 환경에서도 번역이 끊기지 않는 설계
- **STT 이중화**: 한국어는 ReturnZero, 그 외 언어는 Google STT로 분기

### 개발 과정의 도전

- **2개 XR 플랫폼 동시 개발**: 공통 모듈과 플랫폼별 모듈의 경계 설정에 고전 — 중간보고에 문제로 기록하고 구조 조정
- **Cardboard SDK 단종 대응**: 2020년 이후 업데이트 중단된 SDK와 최신 Unity의 호환성 확보

## 4. WPF 마리오 — 엔진 없이 물리·게임 루프 직접 구현 (개인, 2021)

| | |
| --- | --- |
| 기술 | C#, WPF(.NET Framework) — 게임 엔진·물리 라이브러리 미사용 |
| 구현 범위 | 슈퍼 마리오브라더스 1-1 스테이지 |

WPF를 그래픽 출력 용도로만 쓰고, **게임 엔진이 해주던 일을 전부 직접 구현**한 물리 엔진 학습 프로젝트입니다.

- **게임 루프**: 15ms 타이머 기반, 유니티 프레임 흐름을 참고한 Physics() → Update() 실행 순서
- **AABB 충돌 감지** + **터널링(총알 관통) 해결**: 현재 위치→이동 예상 위치의 스윕 영역으로 교차 검사 후 표면 위치 보정
- **겹침 위치 보정**: 겹친 영역의 너비/높이 비교 후 작은 축으로 밀어내기
- **충돌 방향 감지**: 두 객체 원점 간 방향 벡터로 판정
- `GameComponent` 추상 클래스 기반 컴포넌트 구조 (Player/Goomba/Brick/MysteryBox), 클래스 다이어그램 직접 작성

[스크린샷](../assets/mario_1.png) · [충돌 다이어그램](../assets/mario_collision_diagram.png) · [클래스 다이어그램](../assets/mario_class_diagram.png)

4년 뒤 락스텝 동기화(InterplanetaryOnline)와 게임 세션 루프(BattleShip)를 구현할 때의 밑바탕이 된 프로젝트입니다.

## 5. CardFlip — 트윈 연출 데모 (개인, 초단기)

| | |
| --- | --- |
| 기술 | Unity 6 (URP), uGUI/TMP, iTween |
| 저장소 | [CardFlip](https://github.com/Bo-sung/CardFlip) |

**트윈 연출 처리 역량을 보여주기 위한 초단기 소품.** 카드 짝 맞추기(8쌍 16장, 시도 20회)를 소재로:

- iTween 2단계 플립(0→90도에서 앞뒷면 교체 후 90→0도), 결과 패널 easeOutBack 순차 등장
- **애니메이션 중 입력 차단** 처리, 매칭 실패 시 지연 후 자동 되돌리기
- 짧은 제작에도 MonoBehaviour에 의존하지 않는 **순수 C# Presenter(MVP)**로 매칭 판정·상태 로직 분리, 이벤트로 View 통보
